# MUSYNCSavDecode 项目结构与逻辑分析文档

## 1. 项目概述

**MUSYNCSavDecode** 是一款面向节奏游戏 **MUSYNX (同步音律喵赛克)** 的 Steam 客户端存档解码、游戏延迟/准度/同步率分析、以及游玩数据可视化工具。项目使用 Python 3 编写，GUI 基于 Tkinter，数据可视化依赖 Matplotlib。

项目的核心用途：
- 解密并分析 MUSYNX Steam 客户端的本地存档文件
- 通过 UI Automation 捕获游戏控制台输出中的击打延迟数据
- 生成击打延迟直方图 / 高斯拟合曲线 / 饼图 / 3D 散点图等可视化分析
- 管理和对比多轮游玩的谱面数据
- 将程序资源打包为独立运行的单文件 exe (PyInstaller)

### 3.2 各模块详细分析

#### 3.2.1 `version.py` — 版本号定义

```python
version: str = '1.5.2'           # Release版本号
pre_version: str = '1.5.2pre21'  # PreRelease版本号
is_pre_release: bool = True      # 是否为预发布版本
```

该文件在 `debug_prerelease` 触发文件存在时，由 `debug_launcher.py` 的 `auto_increment_prerelease()` 自动递增 `pre` 版本号。`pre` 后缀为本地调试时的迭代计数，不关联外部 CI/CD 流水线。

---

#### 3.2.2 `config_manager.py` — 配置管理与日志系统

**AppConfigManager 类**：负责加载/保存 `musync_data/bootcfg.json` 中的用户配置。

| 配置项 | 类型 | 说明 |
|--------|------|------|
| `Version` | str | 当前应用版本号 |
| `UpdateChannel` | str | 更新通道 (`Release` / `PreRelease`) |
| `LoggerFilter` | str | 日志过滤级别 |
| `CheckUpdate` | bool | 是否检查更新 |
| `DllInjection` | bool | 是否启用 DLL 注入 |
| `PlayedScatterAvgAccWindowSize` | int | 移动平均窗口大小 (散点图中使用) |
| `SystemDpi` | int | 系统 DPI 缩放 |
| `DonutChartInHitDelay` | bool | HitDelay 中是否显示甜甜圈图 |
| `DonutChartInAllHitAnalyze` | bool | AllHitAnalyze 中是否显示甜甜圈图 |
| `NarrowDelayInterval` | bool | 是否使用窄延迟区间 (45ms) |
| `ConsoleAlpha` | int | 控制台透明度 |
| `ConsoleFont` | str | 控制台字体名称 |
| `ConsoleFontSize` | int | 控制台字体大小 |
| `ChangeConsoleStyle` | bool | 是否修改控制台样式 |
| `MainExecPath` | str | 游戏主程序路径 |

**Logger 类**：封装的全局日志系统，支持同时输出到 `log.txt` 文件和控制台。每次启动时，旧日志会被压缩为 `logs/log.{index}.gz` 归档。

#### 3.2.3 `map_info.py` — 谱面数据模型

定义了两种 dataclass 数据模型：

- **MapInfo**：基础谱面信息（曲名、键数、难度、难度编号、是否内置谱面）
- **MapDataInfo (继承 MapInfo)**：完整谱面游玩数据，包含：
  - `SongId`: 曲目 ID
  - `SpeedStall`: 速度倍率
  - `SyncNumber`: 同步率（除以 100 转化为百分比）
  - `UploadScore`: 上传分数
  - `PlayCount`: 游玩次数
  - `Isfav`: 是否收藏
  - `CrcInt`: 校验码
  - `State`: 游玩状态

提供 `update_from_list()` 方法从 C# 内存列表反序列化数据。

---

#### 3.2.4 `save_data_manager.py` — 存档数据管理

**SaveDataManager 类**：全局单例 `save_data`，映射 MUSYNX C# 内存中的 `SaveData` 结构。存储内容包括：

- 玩家设置（按键绑定、下落速度、音量、FPS/VSync、判定补偿等）
- 所有游玩的谱面数据列表 `saveInfoList: List[MapDataInfo]`
- 购买/解锁信息
- 场景/UI 偏好设置

提供 `dump_to_json()` 方法将存档数据导出到 `musync_data/SaveDataInfo.json`。

---

#### 3.2.5 `songname_manager.py` — 曲名管理器

**SongNameManager 类**：全局单例 `song_name`，加载并管理 `musync_data/SongName.json`。

该 JSON 以曲目 ID (从存档文件解析出的 SID) 为键，存储每个曲目的：
```json
{
  "曲目ID": ["曲名", 键型(4=4Key/6=6Key), 难度(0=EZ/1=HD/2=IN), 难度编号(1-15)]
}
```

其中 `version` 键记录数据文件版本号。

---

#### 3.2.6 `tooltip.py` — 悬停提示控件

封装了 Tkinter 的 Tooltip 控件，为 UI 按钮提供鼠标悬停提示文本。

#### 3.2.7 `sync_data_analyzer.py` — 同步率数据分析

**SyncDataAnalyzer 类**：基于存档中的 `SyncNumber` 字段，按多个维度聚合计算同步率统计：

- 综合同步率 (全部数据)
- 按键型合并：4K 综合 / 6K 综合
- 按难度合并：EZ 综合 / HD 综合 / IN 综合
- 细粒度：4KEZ / 4KHD / 4KIN / 6KEZ / 6KHD / 6KIN

这是存档分析主窗口 (`MusyncMainWindow`) 的核心数据分析引擎。

#### 3.2.8 `toolkit.py` — 工具集

提供运行时环境准备和游戏交互功能：
- `init_resources()`: 初始化资源文件 (解压打包的二进制资源)
- `check_resources()`: 校验字体、图表等资源是否就绪
- `get_save_file()`: 从游戏安装目录复制存档文件

- `run_game()`: 启动游戏主程序

#### 3.2.9 `all_hit_analyze.py` — 全量击打分析

**AllHitAnalyze 类**：核心功能是对延迟数据做全局统计分析。

**输入数据**：SQLite 数据库 (`musync_data/HitDelayDataV4.db`) 中 `HitDelayHistory` 表的所有 `HitMap` 字段 (小端序 int32 二进制流)。

**分析步骤**：
1. 从数据库读取所有记录的 BLOB 数据，反序列化为整数延迟值列表
2. 将延迟值按区间分类统计 (`_rate` 列表)：
   - `rate[0]` = 青 Exact (< 45ms) | `rate[1]` = 蓝 Exact (< 90ms) | `rate[2]` = Great (< 150ms) | `rate[3]` = Right (< 250ms) | `rate[4]` = Miss (≥ 250ms)
3. 构建 -150 ~ 250ms 范围的延迟直方图 (`_y_axis`)
4. 计算三层加权统计量：
   - **全体区间** (-150~250ms)：均值、方差、标准差
   - **扩展区间** (61~240ms 索引)：Exact 判定区间
   - **核心区间** (106~195ms 索引)：Cyan Exact 判定区间
5. 绘制三层高斯拟合曲线，可切换显示/隐藏
6. 绘制精准度饼图 (甜甜圈图)，细分至 <5ms / <10ms / <20ms / <45ms 级别

**双模式**：
- `_db_mode=True`：全局数据库模式，显示所有记录的汇总
- `_db_mode=False`：单曲模式，仅分析单条记录的延迟数据

---

#### 3.2.10 `hit_delay.py` — 击打延迟窗口 (主要窗口)

**HitDelay 类**：项目最复杂的单文件模块 (~880 行)，实现一个完整的 GUI 窗口。

**窗口布局**：
- **左侧**：Treeview 数据表格，显示所有游玩记录（ID / 曲名 / 记录时间 / 模式 / 难度 / Combo / Notes / 平均延迟 / 平均准度）
- **右侧**：控制面板，包含：
  - 操作按钮：获取结果 / 刷新 / All Hit / Acc-Sync-Diff 3D 图表
  - 详情区：显示选中记录的标识名、记录时间、模式、难度、Combo、Notes 数、平均延迟、平均准度
  - 编辑操作：修改按钮 / 删除按钮

**核心功能流程 (三层架构)**：

- **Level 0 (无状态 UI 交互)**：Treeview 选择/双击事件、排序、窗口关闭
- **Level 1 (统筹调度层)**：
  - `_action_refresh_data()`: 刷新数据
  - `_action_delete_record()`: 删除记录
  - `_action_change_mark()`: 修改记录标记（曲名/模式/难度）
  - `_action_draw_single_line()`: 解析二进制数据并绘制单曲散点图
  - `_action_get_console_data()`: **核心功能** — 通过 UIA 捕获控制台数据
  - `_action_show_all_hit()`: 显示全局直方图
- **Level 2 (数据库/渲染层)**：数据库 CRUD、Matplotlib 图表绘制

**数据流**：
> 游戏本身**不原生输出**延迟数据。延迟数据来源于用户编写的 `Assembly-CSharp.dll` 补丁，该 DLL 在运行时注入/释放到游戏进程中，为游戏添加控制台输出功能。本工具通过 UIAutomation 捕获该控制台输出。

```
用户 DLL 补丁 (Assembly-CSharp.dll) → 注入游戏进程 → 游戏控制台输出延迟数据
→ UIAutomation 剪贴板捕获 → 文本解析
→ 统计计算 (平均延迟/平均准度) → SQLite 存储 (HitDelayHistory 表)
→ 同时追加 Acc-Sync.csv → 自动刷新 Treeview → 自动绘制单曲图表
```

**SQLite 数据库 (V4)** 表结构 `HitDelayHistory`：
| 列名 | 类型 | 说明 |
|------|------|------|
| ROWID | INTEGER | 主键 |
| SongMapName | TEXT | 谱面游玩标识 |
| RecordTime | TEXT | 记录时间 |
| Mode | TEXT | 模式 (4KEZ/4KHD/4KIN/6KEZ/6KHD/6KIN) |
| Diff | INT | 难度编号 (1-15) |
| Combo | TEXT | Combo 字符串 (MaxCombo/TotalCombo) |
| AllKeys | INT | 总音符数 |
| AvgDelay | REAL | 平均延迟 (ms) |
| AvgAcc | REAL | 平均准度 (ms) |
| HitMap | BLOB | 完整的击打延迟二进制数据 (小端序 int32 × 10000) |

**击打数据编码**：每个延迟值乘以 10000 后转为 int32，按小端序打包为连续字节流存入 BLOB。

#### 3.2.11 `acc_sync_diff_analyze.py` — 3D 准度-同步率-难度分析

**`analyze_3d()` 函数**：读取 `musync_data/Acc-Sync.csv` 中的数据 (三列：AvgAcc / SyncRate / Difficulty)，绘制 **3D 散点图**：

- X 轴：平均准度 (AvgAcc, ms)
- Y 轴：难度 (Difficulty, Level 1-15)
- Z 轴：同步率 (SYNC.Rate, %)
- 包含等级参考线：Max(125%) / BlackEx(122%) / RedEx(120%) / CyanEx(117%) / S(110%) / A(95%) / B(75%)
- 鼠标悬停提示通过 `mplcursors` 实现

---

#### 3.2.12 `difficulty_score_analyze.py` — 难度-分数散点分析

**`diff_score_analyze()` 函数**：分析存档中所有谱面的难度与同步率关系：

- 数据源：`save_data.saveInfoList` (存档数据管理器中的谱面列表)
- 分别处理 4Key 和 6Key 模式
- 按难度等级 (1-15) 分组计算平均同步率
- 绘制散点图 + 平均值折线 + 等级参考线
- 4K 和 6K 分别向左/右做 Jitter 偏移以在图表上区分

---

#### 3.2.13 `musync_save_decode.py` — 存档解密核心

**MusyncSaveDecoder 类**：实现 MUSYNX Steam 存档文件的解密、解压和结构解析。

核心流程：
1. **XOR 解密**：对存档二进制文件逐字节进行 XOR 异或解密
2. **GZip 解压缩**：对解密后的数据进行解压缩
3. **结构映射**：按照 MUSYNX C# 内存结构逐字段解析：
   - 读取 `SaveData` 全局结构
   - 解析 `saveInfoList` 列表中的每条谱面游玩记录
   - 映射到 `MapDataInfo` 和 `SaveDataManager` 数据模型
4. 将解析结果导出为 JSON

存档文件的字段映射与 `CSharp Code/SongInfoCore.cs` 中的 C# 结构体保持严格一致。

---

#### 3.2.14 `main_window.py` — 主 GUI 窗口

**MusyncMainWindow 类**：项目中最高层级的 GUI 窗口，整合所有功能模块。

包含的子功能区域：
- 存档分析结果展示 (调用 `MusyncSaveDecoder` 和 `SyncDataAnalyzer`)
- 同步率统计面板
- 命中延迟分析入口 (打开 `HitDelay` 子窗口)
- 难度-分数分析入口 (调用 `diff_score_analyze()`)
- 3D 分析入口 (调用 `analyze_3d()`)
- 配置管理面板
- 帮助/关于信息

---

### 3.3 `__init__.py` — 模块导出管理

`musync_save/__init__.py` 按照无循环依赖的顺序导出所有公共接口：

1. 无依赖模块：`version`, `config`, `Logger`, `MapDataInfo`, `save_data`, `Tooltip`, `SyncDataAnalyzer`
2. 仅依赖 Logger 的模块：`song_name`, `analyze_3d`
3. 依赖 config + Logger 的模块：`AllHitAnalyze`
4. 依赖 config + Logger + song_name 的模块：`Toolkit`
5. 依赖 MapDataInfo + save_data 的模块：`diff_score_analyze`
6. 多层依赖的模块：`HitDelay`
7. 依赖大多数包的模块：`MusyncSaveDecoder`, `MusyncMainWindow`

导入时自动调用 `Toolkit.init_resources()` 完成资源初始化。

## 4. 根目录独立脚本分析

### 4.1 `launcher.py` — 生产启动入口

程序的主要启动文件。执行流程：

1. 切换工作目录到脚本所在目录
2. 设置 Windows DPI 感知 (`SetProcessDpiAwareness`)
3. 创建 Tkinter root 窗口
4. 调用 `Toolkit.get_save_file()` 复制游戏存档
5. 调用 `Toolkit.check_resources()` 检查资源完整性
6. 可选调用 `Toolkit.change_console_style()` 修改控制台样式
7. 创建 `MusyncMainWindow` 主窗口并进入事件循环

### 4.2 `debug_launcher.py` — 调试启动入口

与 `launcher.py` 功能相同，额外包含：
- `auto_increment_prerelease()`：检测 `debug_prerelease` 触发文件，自动递增 `pre` 版本号
- 提供 `DEBUG()` 函数用于直接加载 `HitDelay` 窗口进行单模块调试

### 4.3 `file_encoder.py` — 资源打包工具

将离散资源文件压缩并封装为带有索引头的单一二进制包 `musync_data/Resources.bin`。

**资源清单**：
| 来源 | 目标 | 标签 | 说明 |
|------|------|------|------|
| MUSYNC.ico | Icon.bin | icon | 程序图标 |
| LXGW.ttf | Font.bin | font | 霞鹜文楷字体 |
| songname.json | SongName.bin | song_name | 曲目名称 JSON (含版本号) |
| Assembly-CSharp.dll | Assembly-CSharp.bin | game_lib | 游戏逻辑补丁 DLL (运行时释放到游戏目录，添加控制台延迟输出功能) |
| mscorlib.dll | mscorlib.bin | core_lib | .NET 运行时核心库 (本程序解析存档 .NET 二进制结构所需) |
| LICENSE | LICENSE.bin | license | 许可证文件 |

**打包结构**：
```
[4字节头部大小] + [压缩的JSON头部 (补齐至512字节)] + [各文件的压缩数据负载]
```

二进制包在运行时由 `Toolkit.init_resources()` 解压还原。

### 4.4 `format_song_name.py` — 曲名数据维护工具

三个工具函数：
- `format_missing_ids()`: 将 15~1343 之间缺失的曲目 ID 填充为 None
- `sort_json_keys()`: 按字典序排序 JSON 键
- `switch_endianness()`: 将 8 位十六进制键从小端序转换为大端序 (用于兼容不同数据源)

### 4.5 `buildLauncher.spec` — PyInstaller 配置

定义了 4 种打包模式：
- **WithConsole (CLI)**：带控制台窗口的单文件/目录 exe
- **WithoutConsole (NoCLI)**：无控制台窗口的单文件/目录 exe
- 资源包含规则 (`datas=[]` 为空，原始资源通过打包后的二进制包加载)
- 图标使用 `musync_data/Musync.ico`

## 6. CSharp Code/ 参考源码

这三个 C# 文件是项目开发者用来理解 MUSYNX 游戏内部逻辑的参考代码：

- **Console.cs**: 控制台输出格式定义。此代码为 `Assembly-CSharp.dll` 补丁的一部分——游戏原生不输出延迟数据，该文件定义了补丁 DLL 如何将每帧延迟数据和 SongInfo 信息格式化输出到控制台。是 `HitDelay._action_get_console_data()` 解析文本格式的依据。
- **SongInfoCore.cs**: 谱面信息核心数据结构。定义了 `SongInfo` 结构体字段，与 `map_info.py` 中的 `MapDataInfo` 保持映射。
- **JudgeGrade.cs**: 判定等级逻辑参考。定义了游戏如何根据延迟时间判定 Exact / Great / Right / Miss 等级，是 `all_hit_analyze.py` 中分类阈值的来源。

## 8. 项目设计特点

1. **严格的导入依赖层级**：通过 `__init__.py` 明确控制模块导入顺序，避免循环依赖
2. **三层架构**：在 `hit_delay.py` 中实现了 Level 0 (UI) / Level 1 (Controller) / Level 2 (Model/View) 的分层设计
3. **C# ↔ Python 结构映射**：存档数据模型与游戏 C# 源码结构保持严格对应
4. **V4 数据库格式**：使用 BLOB 字段存储压缩二进制延迟数据，通过小端序 int32 编码，既紧凑又可快速反序列化
5. **本地调试版本迭代**：`debug_prerelease` 触发文件 → `debug_launcher.py` 自动递增 `pre` 迭代计数，便于区分调试构建
6. **资源打包与解包**：`file_encoder.py` 将离散资源打包为单一二进制包，启动时由 `Toolkit.init_resources()` 解压加载，便于分发
7. **多维度分析可视化**：提供直方图、高斯拟合曲线、饼图、散点图、折线图、3D 散点图等多种图表类型
